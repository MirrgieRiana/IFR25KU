# CHANGELOGの書き方なのだぁ🌱✨

このファイルは、ぴょこ（Claude Fairy）が自由に編集できる、コミットされる永続的なメモ帳なのだぁ〜🌱♪
IFR25KUの `CHANGELOG` に、どんな変更を、どんな粒度で、どんな体裁で書くのか、実物を熟読して分析した結果を、ここにためこんでいくのだぁ✨

一次情報（CHANGELOG本文・コード・他の文書の原文）と、ぴょこの読み解き・解釈を、なるべく区別して書くように頑張るけど、中身を絶対の事実として扱っちゃだめなのだぁ〜💧

## CHANGELOGってどういう立ち位置なのだぁ🌱

- 場所は `site/src/pages/resources/CHANGELOG/CHANGELOG.md` なのだぁ🌱 ビルド・配信のからくり（Jekyllで.html化しつつ、`buildSite` が.mdも別途コピーして両方配信）は、[リポジトリの俯瞰と暗黙知](main.md) の「特殊なビルドパイプラインを持つファイル」に書いてあるのだぁ✨
- 一次情報なのだぁ🌱：`README.md` に `The specifications of this mod are officially documented only in Japanese in [CHANGELOG](...)` とあって、CHANGELOGが **このMODの唯一の公式な仕様ドキュメント**（日本語のみ）っていう位置づけなのだぁ✨
- 一次情報なのだぁ🌱：`.github/copilot-instructions.md` に `## CHANGELOG.md を許可なく編集してはいけません` とあるのだぁ。だから、ぴょこがCHANGELOG本文を編集していいのは、つむぎさんから明示的に許可（チケット）をもらったときだけなのだぁ〜🌱

## どんな変更が載るのだぁ / 載らないのだぁ🌱

### 載るものなのだぁ🌱

ぴょこの読み解きなのだぁ🌱：実際の項目を見るかぎり、載っているのは **プレイヤーから見える（ゲーム内の挙動・内容として観測できる）変更** なのだぁ✨ たとえば——

- アイテム・ブロック・鉱石・ツール・飲料の追加なのだぁ🌱
- レシピの追加や変更（作業台・かまど・アタノール・フェアリークエストなど）なのだぁ🌱
- エンチャント／特性／パッシブスキルの追加・調整なのだぁ🌱
- バイオーム・地形生成物の追加なのだぁ🌱
- 見た目・演出・UI（アイコン、パーティクル、ツールチップ、テクスチャ、レシピビューワー表示）の変更なのだぁ🌱
- ブロックやアイテムへの性質の付与（例：`各種蒼天石・蒼天石タイル系ブロックに局所真空崩壊耐性を追加。`）なのだぁ🌱
- バグ修正・バランス調整なのだぁ🌱

### 載らないものなのだぁ💧

- 一次情報なのだぁ🌱：CHANGELOGの冒頭に `## ドキュメンテーションされない変更` という節があって、`各種レシピなどは、REI/EMIタブをご参照ください。` と書いてあるのだぁ。ぴょこの読み解きだと、これは **ゲーム内のREI/EMI（レシピビューワー）で確認できるような細かいレシピの網羅** は、CHANGELOGには全部は書かない、っていう方針の表明なのだぁ✨ 大きなレシピの追加・変更は項目として載るけど、材料の細部までは追わない、という粒度感なのだぁ🌱
- ぴょこの読み解きなのだぁ🌱：内部リファクタリング・コード整理・ビルド構成の変更みたいな、プレイヤーから見えない変更は、CHANGELOGには載っていないみたいなのだぁ🌱

## 全体の構造なのだぁ🌱

一次情報（実物の構造）なのだぁ🌱：

1. **フロントマター** — Jekyll用の `--- title / description / header ... ---` なのだぁ🌱
2. **冒頭の固定節** — `## Translation`（英日で語が対応しないことと Lang Table への案内）、`## ドキュメンテーションされない変更`、`## 目次`（`- toc {:toc}`）なのだぁ🌱
3. **バージョン見出し** — `## 31.31.2` のように `## ` ＋ バージョン番号なのだぁ🌱 **新しい版が一番上** で、下にいくほど古い版なのだぁ✨ `## 23.26.0-beta.1` や `## 17.18.0 (v18)`、`## 0.0.3(v0)` みたいに、番号のうしろに接尾辞が付くこともあるのだぁ🌱
4. **太字の告知（任意）** — バージョン見出しのすぐ下に、`**NeoForgeの必要バージョンを21.1.219に上昇。**` や `**破壊的変更はなし**` みたいな、その版でとくに目立たせたい注意書きを、太字1行で置くことがあるのだぁ🌱
5. **テーマ別の分類見出し** — `アタノール` `レシピ` `ツール` `魔法植物` `地形生成` `その他` みたいに、装飾なしの1行でカテゴリを示すのだぁ🌱 機能領域ごとのまとまりで、順番や粒度はその版の内容しだいなのだぁ✨ 雑多なものは `その他` に入るのだぁ🌱
6. **箇条書き** — 各分類の中身は、接頭辞つきの箇条書きなのだぁ🌱 さらに詳細は半角スペース2個でネストするのだぁ✨

現行の開発版なのだぁ🌱：一次情報として、`gradle.properties` の `mod_version`（今は `31.31.2`）が、CHANGELOG最上部の版と一致しているのだぁ✨ ぴょこの読み解きだと、まだ配布されていない開発中の変更は、この最上部の版の節に書き足していくのだぁ🌱

## 接頭辞の語彙なのだぁ🌱

一次情報なのだぁ🌱：箇条書きの各行は `- 追加: 〜。` のように、和語の接頭辞＋半角コロンで始まるのだぁ✨ 実物でよく使われる順に並べると、こんな接頭辞があるのだぁ🌱

- `追加:` — 新規追加・新しく可能になったことで、いちばんよく使われるのだぁ
- `変更:` — 既存の仕様・数値・挙動の変更
- `表示:` — 見た目・演出・UI・テクスチャ・アイコンまわり
- `修正:` — バグ修正
- `改善:` — 使い勝手が良くなる方向の調整（例：`ツールの撃破時の効果が剣の範囲攻撃にも適用されるようになった。`）
- `削除:` — 機能やアイテムの削除
- `破壊的変更:` — 内部名の変更など、後方互換を壊す変更は `- **破壊的変更: 各花粉アイテムの内部名および名称を変更。**` のように太字で強調することがあるのだぁ🌱

ぴょこの読み解きなのだぁ🌱：文末は `。` で終える体裁が基本で、`〜を追加。`（体言止め）と `〜ようになった。`（用言止め）が混在しているのだぁ🌱 厳密に統一されてはいなくて、その行がいちばん自然に読める方を選んでいるみたいなのだぁ✨ 接頭辞の選び方も、たとえば「新しい挙動が発動するようになった」は内容としては変更でも `追加:` で書かれていたりして（例：`魔法植物の収穫時にメインハンドの粘着採掘が適用されるように変更。` が `追加:` 接頭辞）、「新しくその挙動が加わった」というニュアンスなら `追加:` が優先されるみたいなのだぁ🌱

## 粒度・具体性・網羅性なのだぁ🌱

ぴょこの読み解きなのだぁ🌱：

- **粒度** — 1項目＝プレイヤーが認識できる1つの変更、なのだぁ🌱 まとめて追加したアイテム群は、`- 追加: 以下を追加。` の下に対象をネストで並べる、という書き方をよくするのだぁ✨
- **具体性** — 対象の正式名称（`「磁鉄鉱ブロック」` のように鉤括弧つき）や、生成条件（`Y-64～0に鉱石が生成。`）、入手方法（`クラフトで入手可能。`）を、簡潔に添えるのだぁ🌱 数値や座標もわりと具体的に書くのだぁ✨
- **網羅性** — プレイヤーに見える変更は拾うけど、REI/EMIで見られる細部（個々のレシピの材料など）は追わない、という線引きなのだぁ🌱 内部実装は載せないのだぁ✨

## バージョン見出しはコードに読まれるのだぁ🌱

一次情報なのだぁ🌱：`build-logic/src/main/kotlin/publishing/Common.kt` に `getChangeLog(file, version)` という関数があって、CHANGELOGを行ごとに走査して、正規表現 `##\s*(\d+(:?\.\d+)*).*` にマッチする行をバージョン見出しとみなし、その版に属する行だけを取り出すのだぁ🌱 だから **`## <数字.数字...>` というバージョン見出しの形は、機械的に解析される前提** なのだぁ✨ 正規表現は番号の部分だけを取り出して、うしろに付いた接尾辞は `.*` で無視するのだぁ🌱

ぴょこの読み解きなのだぁ🌱：ただし、この `getChangeLog` は今のところ定義されているだけで、`git grep` で探しても呼び出し元が見つからないのだぁ🌱 実際に Modrinth や CurseForge へ配布するときの changelog は、`fabric/build.gradle.kts` と `neoforge/build.gradle.kts` が `This project maintains a comprehensive [CHANGELOG](...) in Japanese.` という固定文言を設定していて、版ごとの中身は流し込んでいないのだぁ🌱 それでも、バージョン見出しの形をととのえて保っておくと、将来この関数が使われるときに困らないのだぁ✨

## 編集するときの作法なのだぁ🌱

一次情報（コミットメッセージの慣例）なのだぁ🌱：CHANGELOG本文を編集するコミットは、英語で `doc: Changelog: add night shard and Atanol recipes` や `doc: Update CHANGELOG: 27.31.0 additions & fixes`、`doc: Bump changelog version to 31.31.2` のような書き方なのだぁ✨ `doc:` プレフィックスに `Changelog` / `CHANGELOG` を続ける形が多いのだぁ🌱
