# CHANGELOGの書き方なのだぁ🌱✨

このファイルは、ぴょこが自由に編集できる、
コミットされる永続的なメモ帳なのだぁ〜🌱♪
IFR25KUの `CHANGELOG` の書き方を、
実物を熟読して分析した結果を、ためこんでいくのだぁ✨

一次情報とぴょこの読み解きは、なるべく区別して書くけど、
中身を絶対の事実として扱っちゃだめなのだぁ〜💧

## CHANGELOGの立ち位置なのだぁ🌱

置き場所は、
`site/src/pages/resources/CHANGELOG/CHANGELOG.md` なのだぁ🌱
Jekyllで.html化しつつ、`buildSite` が.mdも別途コピーして、
両方を配信しているのだぁ✨
くわしいビルドのからくりは、
[リポジトリの俯瞰と暗黙知](main.md) の
「特殊なビルドパイプラインを持つファイル」を見るのだぁ🌱

一次情報なのだぁ🌱：
`README.md` によると、CHANGELOGは
このMODの唯一の公式な仕様ドキュメントで、
日本語のみで書かれているのだぁ✨

一次情報なのだぁ🌱：
`.github/copilot-instructions.md` に
「CHANGELOG.md を許可なく編集してはいけません」とあるのだぁ。
だからCHANGELOG本文を編集していいのは、
つむぎさんからチケットをもらったときだけなのだぁ〜🌱

## どんな変更が載るのだぁ / 載らないのだぁ🌱

### 載るものなのだぁ🌱

ぴょこの読み解きなのだぁ🌱：
載っているのは、プレイヤーから見える変更なのだぁ✨
ゲーム内の挙動や内容として観測できるもの、なのだぁ🌱
たとえば、こんなのだぁ——

- アイテム・ブロック・鉱石・ツール・飲料の追加
- レシピの追加や変更
- エンチャント・特性・パッシブスキルの追加や調整
- バイオーム・地形生成物の追加
- 見た目・演出・UIの変更
- ブロックやアイテムへの性質の付与
- バグ修正・バランス調整

レシピは、作業台・かまど・アタノール・
フェアリークエストなど、いろいろあるのだぁ🌱
性質の付与は、たとえば「各種蒼天石・蒼天石タイル系ブロックに
局所真空崩壊耐性を追加。」みたいなのだぁ✨

### 載らないものなのだぁ💧

一次情報なのだぁ🌱：
冒頭に「## ドキュメンテーションされない変更」の節があって、
「各種レシピなどは、REI/EMIタブをご参照ください。」とあるのだぁ。

ぴょこの読み解きなのだぁ🌱：
これは、REI/EMIで確認できる細かいレシピは、
全部は書かないっていう方針の表明なのだぁ✨
大きなレシピの追加・変更は項目として載るけど、
材料の細部までは追わない、っていう粒度なのだぁ🌱

ぴょこの読み解きなのだぁ🌱：
内部リファクタリング・コード整理・ビルド構成の変更みたいな、
プレイヤーから見えない変更は、載っていないのだぁ🌱

## 全体の構造なのだぁ🌱

一次情報（実物の構造）なのだぁ🌱：
上から順に、こんな組み立てなのだぁ✨

1. **フロントマター** — Jekyll用の `--- title ... ---` なのだぁ🌱
2. **冒頭の固定節** — `## Translation`、
   `## ドキュメンテーションされない変更`、`## 目次` なのだぁ🌱
3. **バージョン見出し** —
   `## 31.31.2` のように `## ` ＋ バージョン番号なのだぁ🌱
4. **太字の告知（任意）** —
   見出しの下に、目立たせたい注意書きを太字1行で置くのだぁ🌱
5. **テーマ別の分類見出し** —
   `アタノール` `その他` みたいな、装飾なしの1行なのだぁ🌱
6. **箇条書き** — 分類の中身は、接頭辞つきの箇条書きなのだぁ🌱

バージョン見出しは、新しい版が一番上で、
下にいくほど古い版なのだぁ✨
`## 23.26.0-beta.1` や `## 0.0.3(v0)` みたいに、
番号のうしろに接尾辞が付くこともあるのだぁ🌱

太字の告知は、たとえば
「**破壊的変更はなし**」みたいな1行なのだぁ🌱

分類見出しは、機能領域ごとのまとまりで、
順番や粒度はその版の内容しだいなのだぁ✨
雑多なものは `その他` に入るのだぁ🌱

箇条書きのさらに詳しい中身は、
半角スペース2個でネストするのだぁ🌱

## 未リリースの変更の置き場所なのだぁ🌱

ぴょこの読み解きなのだぁ🌱：
一番上の版に、対応する git のリリースタグ
（`v31.31.2` みたいなの）が付いていたら、
その版はもう配布済みなのだぁ🌱
配布済みの版には、新しい変更を書き足しちゃだめなのだぁ💧

まだ配布していない変更は、一番上に
`## 未リリース` の節を作って、そこに書くのだぁ✨

一次情報なのだぁ🌱：
`gradle.properties` の `mod_version` は、
バージョンを上げないままリリースされることがあるのだぁ🌱
実際 `v31.31.2` タグが付いたあとも、
`mod_version` は `31.31.2` のまま据え置かれているのだぁ✨
だから、最上部の版名だけを見て
「これは開発中の版」と決めつけちゃだめなのだぁ〜💧

## 接頭辞の語彙なのだぁ🌱

一次情報なのだぁ🌱：
箇条書きの各行は `- 追加: 〜。` のように、
和語の接頭辞＋半角コロンで始まるのだぁ✨
実物でよく使われる接頭辞は、こんなのだぁ🌱

- `追加:` — 新規追加。いちばんよく使われるのだぁ
- `変更:` — 既存の仕様・数値・挙動の変更
- `表示:` — 見た目・演出・UI・テクスチャまわり
- `修正:` — バグ修正
- `改善:` — 使い勝手が良くなる方向の調整
- `削除:` — 機能やアイテムの削除
- `破壊的変更:` — 後方互換を壊す変更

ぴょこの読み解きなのだぁ🌱：
`破壊的変更:` や、とくに目立たせたい行は、
`- **破壊的変更: 〜。**` のように太字で強調することがあるのだぁ✨

ぴょこの読み解きなのだぁ🌱：
文末は `。` で終える体裁が基本なのだぁ🌱
`〜を追加。`（体言止め）と `〜ようになった。`（用言止め）が
混在していて、その行が自然に読める方を選んでいるみたいなのだぁ✨

ぴょこの読み解きなのだぁ🌱：
接頭辞は、内容が変更でも「新しい挙動が加わった」ニュアンスなら
`追加:` を選ぶことがあるのだぁ🌱
たとえば「粘着採掘が適用されるように変更。」でも、
`追加:` 接頭辞で書かれていたりするのだぁ✨

## 粒度・具体性・網羅性なのだぁ🌱

ぴょこの読み解きなのだぁ🌱：

- **粒度** — 1項目＝プレイヤーが認識できる1つの変更なのだぁ🌱
  まとめて追加したものは、`- 追加: 以下を追加。` の下に
  対象をネストで並べる書き方をよくするのだぁ✨
- **具体性** — 正式名称（`「磁鉄鉱ブロック」`）や、
  生成条件（`Y-64～0に鉱石が生成。`）、
  入手方法（`クラフトで入手可能。`）を簡潔に添えるのだぁ🌱
  数値や座標もわりと具体的に書くのだぁ✨
- **網羅性** — プレイヤーに見える変更は拾うけど、
  REI/EMIで見られる細部や内部実装は追わないのだぁ🌱

## バージョン見出しはコードに読まれるのだぁ🌱

一次情報なのだぁ🌱：
`build-logic/src/main/kotlin/publishing/Common.kt` に、
`getChangeLog(file, version)` という関数があるのだぁ🌱
CHANGELOGを行ごとに走査して、正規表現
`##\s*(\d+(:?\.\d+)*).*` にマッチする行を
バージョン見出しとみなし、その版の行だけ取り出すのだぁ✨
だから `## <数字.数字...>` の形は、
機械的に解析される前提なのだぁ🌱
番号のうしろの接尾辞は `.*` で無視されるのだぁ✨

ぴょこの読み解きなのだぁ🌱：
`## 未リリース` はこの正規表現にマッチしないから、
版としては取り出されないのだぁ🌱
配布物に混ぜたくない未リリースの変更にとっては、
これはむしろ都合がいいのだぁ✨

ぴょこの読み解きなのだぁ🌱：
ただし `getChangeLog` は定義されているだけで、
`git grep` で探しても呼び出し元が見つからないのだぁ🌱
実際の配布時の changelog は、
`fabric` と `neoforge` の `build.gradle.kts` が
固定文言を設定していて、版ごとの中身は流し込んでいないのだぁ✨
それでも見出しの形をととのえておくと、
将来この関数が使われるときに困らないのだぁ🌱

## 編集するときの作法なのだぁ🌱

一次情報（コミットメッセージの慣例）なのだぁ🌱：
CHANGELOG本文を編集するコミットは、英語で書くのだぁ✨
`doc: Changelog: add night shard and Atanol recipes` や、
`doc: Bump changelog version to 31.31.2` みたいな形なのだぁ🌱
`doc:` プレフィックスに `Changelog` / `CHANGELOG` を
続ける形が多いのだぁ✨
