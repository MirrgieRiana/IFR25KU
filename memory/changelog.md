# CHANGELOGの書き方なのだぁ🌱✨

このファイルは、ぴょこが自由に編集できてコミットされる永続メモなのだぁ〜🌱♪
IFR25KUの `CHANGELOG` の書き方を、実物を熟読して分析したものをためこむのだぁ✨

一次情報とぴょこの読み解きは、なるべく分けて書くのだぁ🌱
でも中身を、絶対の事実として扱っちゃだめなのだぁ〜💧

## CHANGELOGの立ち位置なのだぁ🌱

置き場所は `site/src/pages/resources/CHANGELOG/CHANGELOG.md` なのだぁ🌱
Jekyllで.html化しつつ、`buildSite` が.mdも別途コピーして両方を配信するのだぁ✨
くわしいビルドのからくりは [リポジトリの俯瞰と暗黙知](main.md) にあるのだぁ🌱
その「特殊なビルドパイプラインを持つファイル」の節を見るのだぁ✨

**一次情報**なのだぁ🌱
`README.md` によると、CHANGELOGはこのMOD唯一の公式な仕様書なのだぁ✨
しかも日本語のみで書かれているのだぁ🌱

**一次情報**なのだぁ🌱
`.github/copilot-instructions.md` はCHANGELOGの無断編集を禁じているのだぁ💧
だから本文を編集していいのは、つむぎさんからチケットをもらったときだけなのだぁ🌱

## どんな変更が載るのだぁ / 載らないのだぁ🌱

### 載るものなのだぁ🌱

**ぴょこの読み解き**なのだぁ🌱
載るのは、プレイヤーから見える変更なのだぁ✨
ゲーム内の挙動や内容として、観測できるものなのだぁ🌱
たとえば、こんなのがあるのだぁ——

- アイテム・ブロック・鉱石・ツール・飲料の追加
- レシピの追加や変更
- エンチャント・特性・パッシブスキルの追加や調整
- バイオーム・地形生成物の追加
- 見た目・演出・UIの変更
- ブロックやアイテムへの性質の付与
- バグ修正・バランス調整

レシピは、作業台・かまど・アタノール・フェアリークエストなどがあるのだぁ🌱
性質の付与は「各種蒼天石・蒼天石タイル系ブロックに局所真空崩壊耐性を追加。」みたいなのだぁ✨

### 載らないものなのだぁ💧

**一次情報**なのだぁ🌱
冒頭に「## ドキュメンテーションされない変更」の節があるのだぁ🌱
そこに「各種レシピなどは、REI/EMIタブをご参照ください。」とあるのだぁ✨

**ぴょこの読み解き**なのだぁ🌱
REI/EMIで確認できる細かいレシピは、全部は書かない方針なのだぁ✨
大きなレシピの追加や変更は載るけど、材料の細部までは追わないのだぁ🌱

**ぴょこの読み解き**なのだぁ🌱
内部リファクタリングやビルド構成みたいな、見えない変更は載らないのだぁ🌱

## 全体の構造なのだぁ🌱

**一次情報**（実物の構造）なのだぁ🌱
上から順に、こんな組み立てなのだぁ✨

1. **フロントマター** — Jekyll用の `--- title ... ---` なのだぁ🌱
2. **冒頭の固定節** — `## Translation` などの決まった見出しなのだぁ🌱
3. **バージョン見出し** — `## 31.31.2` のような `## ` ＋ 番号なのだぁ🌱
4. **太字の告知（任意）** — 目立たせたい注意書きを太字1行で置くのだぁ🌱
5. **テーマ別の分類見出し** — `その他` みたいな装飾なしの1行なのだぁ🌱
6. **箇条書き** — 分類の中身は、接頭辞つきの箇条書きなのだぁ🌱

バージョン見出しは、新しい版が一番上で、下にいくほど古い版なのだぁ✨
`## 23.26.0-beta.1` や `## 0.0.3(v0)` みたいに、接尾辞が付くこともあるのだぁ🌱

太字の告知は「**破壊的変更はなし**」みたいな1行なのだぁ🌱

分類見出しは機能領域ごとのまとまりで、雑多なものは `その他` に入るのだぁ✨
順番や粒度は、その版の内容しだいなのだぁ🌱

箇条書きのさらに詳しい中身は、半角スペース2個でネストするのだぁ🌱

## 未リリースの変更の置き場所なのだぁ🌱

**ぴょこの読み解き**なのだぁ🌱
一番上の版にgitのリリースタグが付いていたら、その版は配布済みなのだぁ🌱
配布済みの版には、新しい変更を書き足しちゃだめなのだぁ💧
まだ配布していない変更は、一番上に `## 未リリース` の節を作って書くのだぁ✨

**一次情報**なのだぁ🌱
`gradle.properties` の `mod_version` は、据え置きのままリリースされることがあるのだぁ🌱
実際 `v31.31.2` タグのあとも、値は `31.31.2` のままなのだぁ✨
だから版名だけを見て「開発中の版」と決めつけちゃだめなのだぁ〜💧

## 接頭辞の語彙なのだぁ🌱

**一次情報**なのだぁ🌱
各行は `- 追加: 〜。` のように、和語の接頭辞＋半角コロンで始まるのだぁ✨
実物でよく使う接頭辞は、こんなのがあるのだぁ🌱

- `追加:` — 新規追加。いちばんよく使われるのだぁ
- `変更:` — 既存の仕様・数値・挙動の変更
- `表示:` — 見た目・演出・UI・テクスチャまわり
- `修正:` — バグ修正
- `改善:` — 使い勝手が良くなる方向の調整
- `削除:` — 機能やアイテムの削除
- `破壊的変更:` — 後方互換を壊す変更

**ぴょこの読み解き**なのだぁ🌱
とくに目立たせたい行は `- **破壊的変更: 〜。**` と太字で強調するのだぁ✨

**ぴょこの読み解き**なのだぁ🌱
文末は `。` で終える体裁が基本なのだぁ🌱
体言止めと用言止めが混在していて、すんなり読める方を選ぶみたいなのだぁ✨

**ぴょこの読み解き**なのだぁ🌱
内容が変更でも「新しい挙動が加わった」なら `追加:` を選ぶことがあるのだぁ🌱
粘着採掘の発動も `変更:` ではなく `追加:` なのだぁ✨

## 粒度・具体性・網羅性なのだぁ🌱

**ぴょこの読み解き**なのだぁ🌱

- **粒度** — 1項目＝プレイヤーが認識できる1つの変更なのだぁ🌱
  まとめて足したものは `- 追加: 以下を追加。` の下にネストで並べるのだぁ✨
- **具体性** — 正式名称や入手方法、数値・座標を簡潔に添えるのだぁ🌱
  `Y-64〜0に鉱石が生成。` みたいに、わりと具体的に書くのだぁ✨
- **網羅性** — 見える変更は拾うけど、細部や内部実装は追わないのだぁ🌱

## バージョン見出しはコードに読まれるのだぁ🌱

**一次情報**なのだぁ🌱
`build-logic/src/main/kotlin/publishing/Common.kt` に `getChangeLog(file, version)` があるのだぁ🌱
これはCHANGELOGを行ごとに走査する関数なのだぁ✨
正規表現 `##\s*(\d+(:?\.\d+)*).*` に合う行を、版の見出しとみなすのだぁ🌱
そして、その版の行だけを取り出すのだぁ✨
だから `## <数字.数字...>` の形は、機械的に解析される前提なのだぁ🌱
番号のうしろの接尾辞は `.*` で無視されるのだぁ✨

**ぴょこの読み解き**なのだぁ🌱
`## 未リリース` はこの正規表現に合わないから、版として取り出されないのだぁ🌱
配布物に混ぜたくない未リリースの変更には、むしろ都合がいいのだぁ✨

**ぴょこの読み解き**なのだぁ🌱
ただし `getChangeLog` は定義だけで、呼び出し元が見つからないのだぁ🌱
実際の配布時のchangelogは、`fabric`/`neoforge` の `build.gradle.kts` が固定文言を設定するのだぁ✨
版ごとの中身は、流し込んでいないのだぁ🌱
それでも見出しの形をととのえておくと、将来使われるときに困らないのだぁ✨

## 編集するときの作法なのだぁ🌱

**一次情報**（コミットメッセージの慣例）なのだぁ🌱
CHANGELOG本文を編集するコミットは、英語で書くのだぁ✨
`doc: Changelog: add night shard and Atanol recipes` みたいな形なのだぁ🌱
`doc: Bump changelog version to 31.31.2` みたいな版上げもあるのだぁ✨
`doc:` のあとに `Changelog` / `CHANGELOG` を続ける形が多いのだぁ✨
